# Phase 4 — Approved Patch Application v0 (Workspace Only)

This phase adds the next safe step after a repair proposal is **approved by a
human**: it materializes the approved changes into an **apply workspace**. It
still does **not** modify stable, does not write any real target file, and does
not promote. It builds on Phase 3 (repair proposal-only).

## Why apply still cannot touch stable

Applying a proposal is where an automated system could do real damage. So v0 keeps
the same hard boundary as the proposal phase and adds one capability —
*materializing* the change — but only inside a workspace:

- A human must explicitly approve (marker + named reviewer) **and** pass
  `--approved`; otherwise apply is refused.
- The change is written only under `proposed_changes/` in an apply workspace; the
  intended target files in the repo are never written.
- Nothing is merged or promoted; stable, the safety gate, and the promotion policy
  are untouched.

This lets a human review the *exact* materialized change in one place before
deciding to merge it — without the system ever editing the live tree.

## How an apply workspace is built from an approved proposal

```
approved repair proposal workspace (approval_checklist.md: APPROVED_… + Reviewer)
  → proposal re-validation        (validate_proposal)
  → apply validation              (validate_for_apply: marker + reviewer + apply
                                   allowlist + protected-target block + secret)
  → create apply workspace        proposed_changes/ + apply_manifest.json +
                                   apply_report.md + test_results.json + README
  → record targeted tests         fixed allowlist (executed only on --run-tests)
  → merge / promote               NOT in v0
```

Run it:

```bash
# safe preview — validates, prints the plan, creates no workspace
python scripts/repair_apply.py \
    --proposal-workspace fixtures/repair/fake_approved_proposal_workspace --dry-run

# apply to a candidate apply workspace (requires explicit --approved)
python scripts/repair_apply.py \
    --proposal-workspace fixtures/repair/fake_approved_proposal_workspace \
    --approved --apply-id fake_apply_smoke

python scripts/run_eval.py --task evals/repair/fake_approved_patch_application.yaml   # → 1.0
```

## Targeted tests / regression

The apply records a **fixed, hardcoded** test allowlist (never derived from the
proposal): `validate_structure`, `validate_workflows`, `run_unit_tests`, and the
`vite_login_bug` demo. They are recorded in `test_results.json` by default and
executed only on an explicit `--run-tests`. This is how an apply ties itself to
the existing regression gates without ever running an arbitrary command.

## Remaining risks / limits

- **No merge yet.** The apply workspace is never merged into the repo; a human
  does that separately.
- **No stable promotion yet.** Promotion stays human-driven under the promotion
  policy.
- **proposed_changes still require human review** before any merge.
- **Targeted regression required before merge** — the fixed test allowlist must
  pass.
- **Rollback plan required before promotion** — reversibility of a future merge is
  a later phase.
- **No real API.** Fake provider only.
- **Deterministic fake repair pipeline** — proposals + materialized changes are
  templates, not real diffs generated by a model.

## Next phase (not started)

**Merge + promotion** — a human reviews an apply workspace, merges the proposed
change into a candidate (never stable directly), runs the regression gates, and
only then does the normal promotion policy apply. Merge must keep a rollback and
must never touch stable, the safety gate, or the promotion policy automatically.
